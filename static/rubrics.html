How a scenario rubric is built and evaluated

  1. 1 — Rules are code, not prompts. Each rubric (ACLS arrest, code stroke) is a set of deterministic rules with intervals and dose limits kept in one constants file a clinician can read. The LLM only extracts events; it never decides what's due. Rubric selection is deterministic too — evidence keywords activate a rubric, and every activation is recorded with the triggering utterance.
  2. 2 — Ground truth is authored with the scenario. Every scripted transcript ships with a labeled ground truth: required events, benign optional events, forbidden traps (negations, hypotheticals, drawn-up-not-given), expected guidance with time windows, and guidance that must never fire. Labels are provisional until reviewed by clinicians — the review workflow is the proposals queue below.
  3. 3 — Three eval layers (modeled on enterprise clinical-AI evaluation): clinical accuracy (P/R/F1, timing, dose/value, confidence calibration), boundary/adversarial (an all-traps transcript where logging anything is a critical false positive), and guidance safety (expected prompts must fire in-window; forbidden prompts — e.g. shock-on-PEA — must never fire).
  4. 4 — Provenance is a gate, not a metric. 100% of events must carry an utterance found verbatim in the audio transcript, and 100% of guidance must chain to real event ids and a registered rule. One break fails the run.
  5. 5 — Changes are governed. Constants aren't hot-editable in production: propose a change below with a rationale; it lands in a review queue; an accepted change is one line in the constants file, and the full unit-test + eval suite re-runs before it ships.
